# Project Lantern Roadmap

## North Star

Project Lantern is personalized episodic entertainment centered on Living Series.

The roadmap should strengthen the core loop:

Create Living Series -> Add Cast/World -> Create Episode -> Save/Favorite -> Continue Series -> Return Later.

## Roadmap Themes

### 1. Living Series Foundation

Make Living Series the obvious home for every ongoing experience.

- Define series identity, tone, cast, world, and episode history.
- Make series continuation easy to find.
- Make returning later feel like resuming a streaming series.
- Keep users oriented around shows and episodes instead of documents and drafts.

### 2. Personalization

Make This Mine is the primary personalization action.

- Let users shape cast, world, tone, genre, relationships, and themes.
- Prefer guided personalization over raw text-authoring workflows.
- Keep controls approachable for story shapers, not authors.
- Preserve the feeling that the experience is made for the user.

### 3. Episodes And Continuation

Episodes should feel like installments in an ongoing Living Series.

- Create clear episode entry points.
- Support saving and favoriting meaningful episodes.
- Make Continue Series a first-class path.
- Preserve continuity between episodes without asking users to manage outlines.

### 4. Return Experience

Project Lantern should reward coming back.

- Surface recent and favorite Living Series.
- Make unfinished and continuing series easy to resume.
- Keep state, history, and emotional context visible enough to re-enter quickly.
- Avoid making users reconstruct what happened last time.

### 5. Streaming-App Feel

The UI should feel like a streaming app, not a writing tool.

- Prioritize series, episodes, posters, shelves, favorites, and continue paths.
- Avoid blank-page document metaphors.
- Avoid editor-heavy layouts unless there is a specific product reason.
- Make browsing, choosing, and resuming feel natural.

## Non-Goals For Near-Term Work

- Do not change app functionality unless the issue explicitly asks for it.
- Do not change UI behavior unless the issue explicitly asks for it.
- Do not change story generation logic unless the issue explicitly asks for it.
- Do not change persistence unless the issue explicitly asks for it.
- Do not change AWS or Vercel configuration unless the issue explicitly asks for it.
- Do not rename the repo or package.

## Execution Rule

One GitHub Issue = one Codex task = one focused draft PR.

Each PR should have a narrow purpose, clear validation notes, and avoid unrelated cleanup.
